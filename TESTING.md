# TESTING.md — Test Data + Strategy

> "Take care of testing data" — Sab. This doc is the plan.

## Principles

1. **No customer data in the repo.** Ever. Test inputs are either synthetic or public-domain.
2. **Determinism.** Synthetic PDFs are generated from a seeded `reportlab` script — byte-identical across runs.
3. **No API keys required for unit tests.** Mock backend covers every path.
4. **Golden outputs are versioned.** Once a real backend produces a verified output for a synthetic input, we save it under `tests/fixtures/golden/` and use it forever.
5. **Cheap integration tests.** When an integration test does hit a real API, it does so on the smallest possible image and asserts on a substring — minimal token spend.

## Test data sources

### Synthetic (default, in repo)
Generated by `tests/harness/synthetic_pdf.py` on first run. Three files:

| File | Pages | Content |
|---|---|---|
| `synth_invoice.pdf` | 1 | Invoice header, line items, totals. Known string `INV-2024-001`. |
| `synth_report.pdf` | 2 | Title page + body page with headings and paragraphs. |
| `synth_table.pdf` | 1 | 5×4 numeric table with known cell values. |

Run manually:
```bash
python -m tests.harness.synthetic_pdf --out tests/fixtures
```

### Public PDFs (optional, NOT committed)
For ad-hoc benchmarking the user can drop PDFs into `tests/fixtures/local_only/` (gitignored). The harness will pick them up if present but tests don't depend on them.

### olmOCR-Bench subset (optional)
The eval notebook (`03_evaluation.ipynb`) can download a 10-page subset of the public olmOCR-Bench at run time. The subset is fetched into a gitignored cache; the notebook prints how to skip it offline.

`tests/fixtures/*.pdf` is gitignored (`.gitignore` covers it). Golden outputs ARE committed.

## Markers

```ini
# pyproject.toml [tool.pytest.ini_options]
markers = [
    "integration: requires real API keys; skipped by default",
    "slow: takes > 10s",
    "databricks: requires databricks workspace",
]
```

Default `pytest tests/` runs **unit only** (auto-skips `integration` and `databricks`).

```bash
pytest tests/                    # unit
pytest tests/ -m integration     # integration (needs OPENROUTER_API_KEY)
pytest tests/ -m "not slow"      # fast subset
```

```python
@pytest.fixture(scope="session")

@pytest.fixture
def mock_backend():
    return MockBackend()

@pytest.fixture
def synth_invoice_bytes():
    return (FIXTURES / "synth_invoice.pdf").read_bytes()
```

## Golden test pattern

```python
def test_processor_against_golden(spark, mock_backend, synth_invoice_bytes):
    pipeline = OCRPipeline(backend="mock", ...)
    df = pipeline.parse_single(synth_invoice_bytes)
    assert_golden_md(df["markdown"][0], golden_name="synth_invoice")
```

`assert_golden_md` does a normalized edit-distance comparison and tolerates a small delta (`<= 0.05`).

## Harness engineering (what to extend)

The harness has three jobs:
1. **Generate** deterministic synthetic data.
2. **Assert** outputs against golden files.
3. **Benchmark** throughput, latency, and cost.

To add a new synthetic doc type:
1. Add a builder method on `SyntheticPDFBuilder` in `tests/harness/synthetic_pdf.py`.
2. Add the golden in `tests/fixtures/golden/<name>.md`.
3. Add an assertion test in `tests/test_processor.py`.

To extend the perf harness:
1. Edit `tests/harness/perf.py::run_perf`.
2. New metrics get appended to `runtime/BENCH.md` via `doc-scribe` agent.

See `HARNESS.md` for the harness internals.

GitHub Actions example (not committed by default — add when ready):

```yaml
# .github/workflows/test.yml (NOT committed yet)

jobs:
  unit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5

      - uses: actions/setup-java@v4
        with: { java-version: "17", distribution: "temurin" }
      - run: pip install uv && uv sync --extra dev

```

Integration tests deliberately stay off CI to avoid leaking the API key — they're a local-only safety net.
